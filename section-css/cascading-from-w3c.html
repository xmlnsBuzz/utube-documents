1. 소개

CSS의 근본적인 설계 원리 중 하나는 계단식으로 , 여러 스타일 시트가 문서의 표현에 영향을 미칠 수 있습니다. 다른 선언이 동일한 요소 / 속성 조합에 대한 값을 설정하려고하면 충돌이 어떻게 든 해결되어야합니다.

반대의 문제는 선언이 요소 / 속성 조합의 값을 설정하려고 시도하지 않을 때 발생합니다. 이 경우, 상속을 통해 또는 속성의 초기 값 을보고 값 을 찾습니다 .

연계 및 디폴트 처리는 입력으로 선언 세트를 취득하고, 출력 지정된 값을 각 요소에 대한 각각의 속성을.

문서의 모든 요소에 대한 모든 속성에 대해 지정된 값을 찾는 규칙이이 사양에 설명되어 있습니다. 페이지 컨텍스트와 여백 상자에서 지정된 값을 찾는 규칙은 [css-page-3]에 설명되어 있습니다.

1.1. 모듈 상호 작용

이 절은 규범 적이다.

이 모듈은 [CSS2] 6 장 에서 정의 된 속성 값, 계단식 및 상속을 할당하는 규칙을 대체하고 확장합니다 .

다른 CSS 모듈은 여기에 정의 된 일부 구문 및 기능의 정의를 확장 할 수 있습니다. 예를 들어, Media Queries Level 4 사양을이 모듈과 결합하면이 사양에서 사용 된 <media-query> 값 유형 의 정의가 확장됩니다 .

2. 스타일 시트 가져 오기 : @import 규칙

@import의 규칙은 사용자가 다른 스타일 시트에서 스타일 규칙을 가져올 수 있습니다. 는 IF @import의 규칙이 유효한 스타일 시트를 참조, 사용자 에이전트는이 자리에 기록 된 것처럼 스타일 시트의 내용을 취급해야 @import의 두 가지 예외를 제외하고, 규칙을 :

  • @namespace 규칙 과 같은 기능 이 특정 스타일 시트에만 적용되고 가져온 속성 은 적용되지 않는다고 명시 적으로 정의한 경우 가져온 스타일 시트에는 적용되지 않습니다.

  • 기능이 스타일 시트의 두 개 이상의 구문의 상대적인 순서에 의존하는 경우 (예 : @charset 앞에 다른 내용이 없어야 함) 동일한 스타일 시트의 구문간에 만 적용됩니다.

예를 들어, 가져온 스타일 시트의 스타일 규칙에있는 선언은 @import 의 시점에서 스타일 시트에 글자 그대로 쓰여진 것처럼 캐스 캐 이드와 상호 작용합니다 .

모든 @import 규칙은 스타일 시트 ( @charset 외에 스타일 시트 의 첫 번째 것이어야 함)에있는 다른 모든 규칙 및 스타일 규칙보다 우선해야합니다. 그렇지 않으면 @import 규칙이 유효하지 않습니다. @import 의 구문 은 다음과 같습니다.

@import [ <url>  |  <string> ] <media-query-list> ? ;

를 Where <URL> 또는 <문자열>는 스타일 시트의 URL을 제공합니다 수입하고 선택합니다 <미디어 쿼리 목록> 합니다 ( 수입 조건 )이 적용되는 조건을 말한다.

경우 <문자열> 제공, 그것은으로 해석해야 <URL> 같은 값.

다음 줄은 의미가 동일하며 @import 구문 ( url () 과 벌크 문자열 모두)을 보여줍니다. 
@import  "mystyle.css" ; 
  @import  url ( "mystyle.css" );

수입 조건은 수입이 매체에 의존 할 수 있습니다. 어떤이 없을 수입 조건 , 수입은 무조건입니다. ( <media-query-list>에 all 을 지정해도 동일한 효과가 있습니다.)

URL 다음의 표현식 평가 및 전체 구문은 미디어 쿼리 사양 [MEDIAQ]에 의해 정의됩니다 . 미디어 쿼리가 일치하지 않으면 가져온 스타일 시트의 규칙이 가져온 스타일 시트가 지정된 미디어 쿼리로 @media 블록에 래핑 된 것처럼 적용되지 않습니다 . 따라서 사용자 에이전트는 미디어 쿼리가 일치하지 않는 한 미디어 종속 가져 오기를 가져 오는 것을 피할 수 있습니다.

다음 규칙은 @import 규칙을 미디어 종속적으로 만들 수있는 방법을 보여줍니다 . 
@import  url ( "fineprint.css" ) print ; 
  @import  url ( "bluish.css" ) 투영 , tv ; 
  @import  url ( "narrow.css" ) 핸드 헬드 및 ( 최대 너비 : 400px);

동일한 스타일 시트가 여러 위치에서 문서로 가져 오거나 링크되는 경우, 사용 도구는 링크가 독립적 인 스타일 시트에있는 것처럼 각 링크를 처리해야합니다.

참고 : 이는 리소스 가져 오기에 대한 요구 사항을 제시하지 않으며 스타일 시트가 CSSOM에 반영되고이 스타일 시트와 같은 사양에 사용되는 방법 만 포함합니다. 적절한 캐싱을 가정 할 때, UA가 스타일 시트를 여러 번 링크하거나 가져 왔더라도 한 번만 가져올 수 있습니다.

가져온 스타일 시트 의 원본 은 가져온 스타일 시트의 원본 입니다.

가져온 스타일 시트 의 환경 인코딩 은 가져온 스타일 시트의 인코딩입니다. [css-syntax-3]

2.1. CSS 스타일 시트의 내용 유형

가져온 스타일 시트 처리는 링크 된 자원의 실제 유형에 따라 다릅니다. 리소스에 Content-Type 메타 데이터 가 없거나 호스트 문서가 쿼크 모드에 있고 가져온 스타일 시트 와 같은 출처 를 가진 경우 연결된 리소스의 유형은 다음과 같습니다 text/css. 그렇지 않으면 유형은 Content-Type 메타 데이터 에서 결정됩니다 .

링크 된 자원 유형이 text/css이면, CSS 스타일 시트로 해석되어야합니다. 그렇지 않으면 네트워크 오류로 해석되어야합니다.

3. 속기 속성

일부 속성은 속기 속성 이므로 작성자는 속성 하나를 사용하여 여러 속성의 값을 지정할 수 있습니다. 약식 속성은 그 모든 설정 긴 형식 하위 속성을 대신 확장 정확히 것처럼.

속기 양식 에서 값을 생략 할 경우 달리 정의되지 않는 한 각 "누락 된" 하위 특성 에는 초기 값 이 지정됩니다 .

즉, 속기 속성 선언 은 명시 적으로 설정되지 않은 하위 속성 을 포함하여 모든 하위 속성을 항상 설정 합니다 . 부주의하게 사용하면 실수로 일부 하위 속성이 재설정 될 수 있습니다 . 조심스럽게 사용되면 속기 는 부주의로 다른 소스에서 계단식으로 하위 속성 을 재설정하여 "빈 슬레이트"를 보장 할 수 있습니다.

예를 들어 배경색 : 녹색이 아닌 배경색 : 녹색을 작성 하면 배경색이 배경 이미지로 배경을 이미지로 설정했을 수있는 이전 선언을 무시합니다 .

예를 들어 CSS 레벨 1 글꼴 속성은 글꼴 스타일 , 글꼴 변형 , 글꼴 두께 , 글꼴 크기 , 선 높이 및 글꼴 모음 을 한꺼번에 설정하기위한 속기 속성입니다 . 이 예제의 여러 선언은 다음과 같습니다. 
h1 { 
    글꼴 두께 : 굵게 ; 
    폰트 크기 :  12 에서 pt ; 
    라인 높이 :  14 에서 pt ; 
    font-family : Helvetica ; 
    글꼴 변형 : 보통 ; 
    글꼴 스타일 : 일반 ; 
  }

따라서 다음과 같이 다시 쓸 수 있습니다.

H1 {  폰트 : 굵은 12 에서 pt / 14 에서 pt 돋움 }

더 많은 글꼴 하위 속성 이 CSS에 도입되면 속기 선언은 그것들을 초기 값으로 재설정합니다.

경우에 따라 속기 는 하위 속성의 값과 직접적으로 일치하지 않는 구문 또는 특수 키워드가 다를 수 있습니다 . 이러한 경우, 속기 는 명시 적으로 값의 확장을 정의합니다.

다른 경우에, 속성은있을 리 전용 서브 속성 기타 같은 속기의 하위 속성 , 상기 속기 의해 때 불특정 초기 값으로 리셋하지만 속기는 설정 구문이 포함되지 않을 수 서브을 속성 을 다른 값으로 변경합니다. 예를 들어 border shorthand는 border-image 를 none 의 초기 값으로 재설정 하지만 다른 것으로 설정하는 구문은 없습니다. [css-backgrounds-3]

경우 속기가 의 하나로서 지정된 CSS 전체 키워드 [CSS - 값 3] , 그것의 모든 세트 하위 속성 임을 포함한 모든 해당 키워드에 리셋 전용 서브 속성 . (이 키워드는 단 하나의 선언에있는 다른 값과 결합 될 수 없으며, 속기에는 포함되지 않습니다.)

선언 속기 될 속성을 ! 중요한 것은 그 모든 선언에 해당 하위 속성을 로 ! 중요합니다 .

3.1. 모든 속성 재설정 : all 속성

4. 가치 처리

사용자 에이전트가 문서를 파싱하고 문서 트리를 구성한 후에는 트리의 모든 요소와 서식 구조의 모든 상자에 대상 미디어 유형에 적용되는 모든 속성의 값을 할당해야합니다.

주어진 요소 또는 상자에 대한 CSS 속성의 최종 값은 다중 단계 계산의 결과입니다.

  1. 먼저 요소에 적용된 선언 된 모든 값 이 각 요소의 각 속성에 대해 수집됩니다. 요소에 0 또는 여러 개의 선언 된 값이 적용될 수 있습니다 .
  2. 계단식 배열은 계단식 값을 산출합니다 . 요소 당 속성 당 최대 하나의 계단식 값이 있습니다.
  3. Defaulting은 지정된 값을 산출합니다 . 모든 요소에는 속성 당 정확히 하나의 지정된 값 이 있습니다.
  4. 값 종속성을 해결하면 계산 된 값 이 산출됩니다 . 모든 요소에는 속성 당 정확히 하나의 계산 된 값 이 있습니다.
  5. 문서 서식을 지정하면 사용 된 값이 산출 됩니다 . 해당 속성이 요소에 적용되는 경우 요소는 지정된 속성에 대해 사용 된 값만 갖습니다 .
  6. 마지막으로, 사용 된 값은 디스플레이 환경의 제약 조건에 따라 실제 값 으로 변환됩니다 . 사용 된 값 과 마찬가지로 요소의 특정 속성에 대한 실제 값 이 있거나 없을 수 있습니다 .

4.5. 사용 된 값

사용 값은 복용의 결과 계산 된 값 과, 문서의 레이아웃에 사용 절대 이론치하도록 나머지 계산을 완료한다. 속성이이 요소에 적용되지 않으면 해당 요소 에 해당 값 이 사용 되지 않습니다 .

예를 들어 width : auto 선언은 요소 조상의 레이아웃을 알지 못하는 길이로 해석 될 수 없으므로 계산 된 값 은 자동 이며 사용 된 값 은 절대 길이 (예 : 100px) 입니다. [CSS2]

또 다른 예로, a <div>는 계산 된 break-before 값 auto를 가질 수 있지만 첫 번째 하위에서 propagation by page 의 사용 된 break-before 값을 얻습니다 . [css-break-3]

마지막으로 속성이 요소에 적용되지 않으면 사용 된 값 이 없습니다 . 예를 들어, flex 속성에는 유연한 항목 이 아닌 요소에 사용 된 값 이 없습니다 .

4.6. 실제 값

사용되는 값은 원칙적으로 사용할 준비가 있지만, 사용자 에이전트는 주어진 환경에서 값을 활용하지 못할 수 있습니다. 예를 들어 사용자 에이전트는 정수 픽셀 너비의 테두리 만 렌더링 할 수 있으므로 사용 된 너비 와 비슷해야합니다 . 또한 요소의 글꼴 크기는 글꼴의 사용 가능성 또는 font-size-adjust 속성 의 값에 따라 조정해야 할 수 있습니다. 실제 값은 그러한 조정이 이루어진 후 사용 된 값입니다.

참고 : 요소의 실제 값을 프로빙하면 문서 배치 방법에 대해 많이 알 수 있습니다. 그러나 모든 정보가 실제 값으로 기록되는 것은 아닙니다. 예를 들어, page-break-after 속성 의 실제 값은 요소 다음에 페이지 나누기가 있는지 여부를 반영하지 않습니다. 마찬가지로, 고아 의 실제 값은 특정 요소에 고아 라인이 얼마나 많은지 반영하지 않습니다. 아래 표의 예 (j)와 (k)를 참조하십시오 .

5. 필터링

선언 된 값 을 찾으려면 구현은 먼저 각 요소에 적용되는 모든 선언을 식별해야합니다. 다음 경우에 선언이 요소에 적용됩니다.

  • 현재이 문서에 적용되는 스타일 시트에 속합니다.
  • 거짓 조건 이있는 조건부 규칙 [CSS3-CONDITIONAL]에 의해 규정되지 않습니다 .
  • 선택기가 요소와 일치하는 스타일 규칙에 속합니다. [SELECT] ( 필요하다면 스코핑 을 고려하십시오.)
  • 문법적으로 유효합니다. 선언의 속성은 알려진 속성 이름이고 선언의 값은 해당 속성의 구문과 일치합니다.

적용되는 선언 값은 각 요소의 각 특성에 대해 선언 된 값 목록을 형성 합니다 . 다음 절인 캐스케이드 는이 목록의 우선 순위를 지정합니다.

  • 스타일 속성의 선언은 스타일 속성이 표시되는 요소의 문서 순서에 따라 정렬되며 모든 스타일 시트 뒤에 배치됩니다.

6.1. 계단식 기원

각 스타일 규칙에는 계단식 입력란 이 있으며 계단식 입력란이 계단식으로 들어오는 위치를 결정합니다. CSS는 세 가지 핵심 기원을 정의합니다 .

작성자 원산지
작성자는 문서 언어 규칙에 따라 원본 문서의 스타일 시트를 지정합니다. 예를 들어 HTML에서 스타일 시트는 문서에 포함되거나 외부 적으로 링크 될 수 있습니다.
사용자 원산지
사용자는 특정 문서에 대한 스타일 정보를 지정할 수 있습니다. 예를 들어, 사용자는 스타일 시트를 포함하는 파일을 지정하거나 사용자 에이전트가 사용자 스타일 시트를 생성하는 인터페이스를 제공 할 수 있습니다.
사용자 에이전트 원본
적합한 사용자 에이전트는 기본 스타일 시트를 적용해야합니다 (또는 마치 마치 것처럼 행동해야합니다). 사용자 에이전트의 기본 스타일 시트는 문서 언어에 대한 일반적인 표현 기대치를 만족시키는 방식으로 문서 언어 요소를 제공해야합니다 (예 : 시각적 브라우저의 경우 HTML의 EM 요소는 기울임 꼴 글꼴로 표시됨). 예를 들어 HTML 사용자 에이전트 스타일 시트를 참조하십시오 . [HTML]

CSS는 작성자와 사용자 스타일 시트간에 균형을 유지하려고 시도합니다. 기본적으로 작성자의 스타일 시트에있는 규칙은 사용자 에이전트의 기본 스타일 시트에있는 규칙을 덮어 쓰는 사용자의 스타일 시트에있는 규칙보다 우선합니다. 이 균형을 유지하기 위해 선언을 중요 하게 만들 수 있습니다. 선언은 계단식으로 가중치를 높이고 우선 순위를 반전시킵니다.

6.3. CSS가 아닌 프레젠테이션 힌트의 우선 순위

UA는 소스 문서의 마크 업 (예 : [HTML] 의 bgcolor속성 또는 s요소)에서 프리젠 테이션 힌트를 존중할 수 있습니다 . 모든 문서 언어 기반 스타일링은 해당 CSS 규칙으로 변환되어야하며 사용자 에이전트 수준에서 캐스케이드를 입력하거나 작성자 스타일 시트의 시작 부분에 특수성이 0 인 작성자 수준 규칙으로 취급되어야합니다. 문서 언어는 프리젠 테이션 힌트가 UA 또는 캐스케이드의 작성자 레벨에 입력되는지 여부를 정의 할 수 있습니다. 그렇다면 UA는 그에 따라 행동해야합니다. 예를 들어 [SVG11] 은 프리젠 테이션 속성을 작성자 레벨에 매핑합니다.

참고 : UA 수준에서 캐스케이드로 들어가는 프리젠 테이션 힌트는 작성자 또는 사용자 스타일에 의해 무시 될 수 있습니다. 작성자 수준에서 캐스케이드에 들어가는 프리젠 테이션 힌트는 작성자 스타일에 의해 무시 될 수 있지만 중요 하지 않은 사용자 스타일에 의해 무시 될 수 있습니다 . 호스트 언어는 이러한 고려 사항을 염두에두고 프리젠 테이션 힌트에 적절한 수준을 선택해야합니다.

7. 채무 불이행

때 계단식 값 발생하지 않습니다의 지정된 값은 다른 방법을 발견해야합니다. 상속 된 속성 은 상속을 통해 부모 요소에서 기본값을 가져 옵니다 . 다른 모든 속성은 초기 값을가집니다 . 제작자는 inherit 및 initial 키워드 를 통해 상속 또는 초기화를 명시 적으로 요청할 수 있습니다 .

7.1. 초기 값

각 속성에는 속성 정의 테이블에 정의 된 초기 값이 있습니다. 속성이 없으면 상속 된 속성 과 캐스케이드 값을 초래하지 않고, 그 지정된 값 속성의 그것 인 초기 값 .

7.2. 계승

상속 은 부모 요소에서 자식 요소로 속성 값을 전달합니다. 요소의 상속 된 값 은 요소의 부모 요소에있는 속성 의 계산 된 값 입니다. 상위 요소가없는 루트 요소의 경우 상속 된 값 은 속성 의 초기 값 입니다.

(의사 요소는 각 의사 요소 [SELECT] 에 대해 설명 된 가상의 태그 시퀀스에 따라 상속됩니다 .)

일부 속성은 속성 정의 테이블에 정의 된대로 상속 된 속성 입니다. 이것은 캐스케이드 결과가 값이 아닌 경우 값이 상속에 의해 결정된다는 것을 의미합니다.

속성은 명시 적으로 상속 될 수도 있습니다. inherit 키워드를 참조하십시오 .

참고 : 상속은 문서 트리를 따르며 익명 상자에 의해 차단 되거나 그렇지 않으면 상자 트리의 조작에 의해 영향을받지 않습니다.

스타일 시트는이 모듈에 정의 된 구문을 사용하는 모든 명령문이이 모듈에 정의 된 각 기능의 일반 CSS 문법 및 개별 문법에 따라 유효한 경우이 사양을 준수합니다.

렌더러는, 적절한 사양으로 정의 된 스타일 시트를 해석하는 것 외에도, 올바르게 스펙 분석하고 그에 따라 문서를 렌더링함으로써이 스펙이 정의한 모든 기능을 지원하는 경우,이 스펙을 준수합니다. 그러나 UA가 장치의 한계로 인해 문서를 올바르게 렌더링 할 수 없다고해서 UA가 부적합하게되지는 않습니다. 예를 들어, UA는 흑백 모니터에서 색상을 렌더링 할 필요가 없습니다.

저작 도구는이 모듈의 일반 CSS 문법 및 각 기능의 개별 문법에 따라 구문 적으로 올바른 스타일 시트를 작성하고이 모듈에 설명 된대로 스타일 시트의 다른 모든 적합성 요구 사항을 충족하면이 사양을 준수합니다.

CSS의 책임 구현을위한 요구 사항

다음 섹션에서는 현재와 미래의 상호 운용성을 향상시키는 방식으로 책임있게 CSS를 구현하기위한 몇 가지 준수 요구 사항을 정의합니다.

부분 구현

제작자가 순방향 호환 파싱 규칙을 활용하여 폴백 값을 할당 할 수 있도록 CSS 렌더러 는 사용 가능한 규칙이없는 규칙, 속성, 속성 값, 키워드 및 기타 구문 구문 을 무효로 처리 하고 적절하게 무시 해야합니다 지원 수준 . 특히 사용 도구 는 지원되지 않는 속성 값을 선택적으로 무시 해서는 안되며 지원되는 값을 단일 다중 값 속성 선언에서 준수해야합니다. 즉, 지원되지 않는 값을 사용해야하는 값이 유효하지 않은 것으로 판단되면 CSS는 전체 선언을 무시해야합니다.

불안정하고 독점적 인 기능의 구현

향후 안정적인 CSS 기능과의 충돌을 피하기 위해 CSSWG 는 불안정한 기능 및 CSS에 대한 독점 확장 구현에 대한 모범 사례 를 따르는 것이 좋습니다 .

CR 종료 기준

이 명세를 권고안으로 개선하기 위해서는 각 기능에 대해 최소한 두 개의 독립적이고 상호 운용 가능한 구현이 있어야한다. 각 기능은 다른 제품 세트로 구현 될 수 있지만 모든 기능을 단일 제품으로 구현할 필요는 없습니다. 이 기준의 목적을 위해 다음 용어를 정의합니다.

독립적 인
각 구현은 다른 당사자에 의해 개발되어야하며 다른 적격 구현에 의해 사용 된 코드를 공유, 재사용 또는 파생 할 수 없습니다. 이 명세의 구현에 아무런 영향을 미치지 않는 코드 섹션은이 요구 사항에서 제외됩니다.
상호 운용 가능한
공식 CSS 테스트 스위트에서 각각의 테스트 케이스를 통과 시키거나, 구현이 웹 브라우저가 아니면 동등한 테스트를 통과해야한다. 테스트 스위트의 모든 관련 테스트는 그러한 사용자 에이전트 (UA)가 상호 운용성을 주장하는 데 사용되는 경우 동등한 테스트를 만들어야합니다. 또한 그러한 UA가 상호 운용성을 주장하는 데 사용되는 경우 상호 운용성을 목적으로 동일한 테스트를 통과 할 수있는 하나 이상의 추가 UA가 있어야합니다. 동등한 시험은 동료 심사를 위해 공개적으로 이용 가능해야한다.
이행
사용자 에이전트 :
  1. 사양을 구현합니다.
  2. 일반인이 이용할 수 있습니다. 구현은 선적 제품 또는 공개적으로 사용 가능한 다른 버전 (예 : 베타 버전, 미리보기 릴리스 또는 "야간 빌드") 일 수 있습니다. 비 운송 제품 릴리스는 안정성을 입증하기 위해 최소 한 달 동안 기능을 구현해야합니다.
  3. 실험적이지 않습니다 (즉, 테스트 슈트를 통과하도록 특별히 고안된 버전이며 정상적인 용도로는 사용되지 않습니다).

사양은 최소 6 개월 동안 후보 추천으로 남습니다.
